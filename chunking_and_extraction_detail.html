Chunking strategy and feature extraction detail for NCRB RAG pipeline

THE CORE CHALLENGE — WHY NAIVE CHUNKING FAILS ON NCRB
Problem 1: Tables split across PDF pages lose row context
Table 1A.4 spans multiple pages. A naive 512-token splitter will cut mid-table, creating chunks where "Rajasthan ... 312.4" appears without the column header "IPC crime rate per lakh". The retriever finds the number but not what it means.
WRONG — naive chunk
"...Bihar 287.3\nRajasthan 312.4\nUP 198.6\nWB 201.1..."
RIGHT — row-aware chunk
"Table 1A.1 | State: Rajasthan | Metric: IPC crime rate per lakh population | Year: 2022 | Value: 312.4"
Problem 2: Multi-column tables need column header injection
NCRB tables have 8–12 columns. pdfplumber extracts them spatially — but if you chunk a row without its headers, the LLM cannot map column position to metric name. Solution: prepend column headers to every row chunk during ingestion.
RAG retrieves this
"Kerala | 96.0 | 87.2 | 41.3 | 12.8 | ..."
Should retrieve this
"State: Kerala | Charge-sheeting rate: 96.0% | Court disposal rate: 87.2% | Conviction rate: 41.3% | Acquittal rate: 12.8%"
CHUNKING STRATEGY — TWO SEPARATE INDEXES
Index A — Structured table index
Each row of each table becomes one document. Metadata: table_id, state, year, metric_names, values. Queried with exact filters — "retrieve all rows where state=Kerala". Fast, precise numeric lookup.
For feature extraction
Index B — Prose + snapshot index
Snapshot text, chapter summaries, methodology sections chunked at sentence level with 50-token overlap. Semantic search. Used for explanatory QA — "why did cyber crime rise?"
For user QA
Query router
Classify each query as "numeric lookup" (→ Index A, exact retrieval) or "explanatory" (→ Index B, semantic search). A simple intent classifier or keyword check handles this.

FEATURE EXTRACTION — HOW TO MAKE IT RELIABLE
Extraction pipeline per state per feature (Step 06 detail)
Do not rely on the LLM to extract numbers from free text — LLMs hallucinate numeric values. Instead, use a two-stage approach: RAG retrieves the correct chunk, then a deterministic regex extracts the number from that chunk. The LLM is only used for routing and answer generation, never for number production.
STAGE 1 — RAG retrieves
Query: "Charge-sheeting rate for Kerala, Table 19A, 2022"

Retrieved chunk: "State: Kerala | Charge-sheeting rate: 96.0% | Table: 19A | Year: 2022"
STAGE 2 — Regex extracts
pattern = r"Charge-sheeting rate:\s*([\d.]+)%"

match → 96.0
type → float
unit → percent
valid range → [0, 100] ✓
HANDLING MISSING / FAILED EXTRACTIONS
Tier 1 — RAG + regex
Primary path. Succeeds for well-structured table rows with clean pdfplumber extraction.
Tier 2 — Direct table parse
If RAG fails, fall back to direct pdfplumber table extraction for known page ranges. No vector search.
Tier 3 — Imputation
If both fail (e.g. Nagaland data gap flagged in report), impute with regional median or flag as missing in feature matrix.
